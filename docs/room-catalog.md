# Room Type Catalog — 50 Workshop Rooms

Each project in the workshop is assigned one of these 50 room types on first-add. Assignment is random but permanent (`room_type` stored in `projects.json` once chosen).

The variety here serves cognitive anchoring: distinct visual character per project gives Karl's mind a place to "go" when switching contexts.

---

## How rooms are described

Each entry has:
- **id** — kebab-case identifier used in `projects.json`
- **name** — display name
- **vibe** — one-line aesthetic summary
- **materials** — dominant surface materials for procedural + texture rendering
- **lighting** — how the room is lit
- **mood** — emotional/cognitive tone
- **whiteboard treatment** — where data displays live in this room

## Assignment algorithm (deterministic-on-first-add)

```js
// When a project first appears in projects.json without a room_type:
const projectId = project.id;
const seed = hashString(projectId);  // deterministic from ID
const roomTypes = catalog.length;     // 50
const chosen = catalog[seed % roomTypes];
project.room_type = chosen.id;
// Write back to projects.json. Never re-roll.
```

The `hashString` is a stable hash (e.g., FNV-1a or similar) so a given project ID always yields the same seed and therefore the same room. This makes assignment reproducible without storing state, though we DO store the `room_type` field so manual overrides survive.

## Manual override

If Karl wants a specific project to use a specific room type (e.g., "Solo Orchestrator should be the code-lab"), just edit `projects.json` to set `room_type: "code-lab"`. The renderer reads the stored value before falling back to assignment.

## Building rooms incrementally

The 20 rooms built initially should cover the 14 floor stations plus buffer for the 4 annex stations when they graduate. The remaining 30 rooms get built when projects need them — adding a new project triggers either a build of its assigned room (if not yet built) or use of the existing build (if previously built for another project).

This is fine because room types are catalog entries, not per-project bespoke builds — `industrial-machine-shop` is one Three.js scene that any project assigned to it renders into. The catalog is shared; the data populating the whiteboards is per-project.
